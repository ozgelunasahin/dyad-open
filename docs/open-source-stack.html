Open source stack
for content moderation
and governance.

Six platforms assessed for fork-and-implement suitability. Two dimensions: automated + community moderation, and participatory governance. Recommendations included.

At a glance

Six platforms. Two questions.

Can we fork it for content moderation? Can we fork it for community governance? Every platform below is assessed on those two axes, with a recommendation on whether to fork, integrate via API, borrow the design pattern, or skip.

Content moderation

Fork LocalMod

Python REST API. Fully self-hosted. Toxicity, NSFW text and image, PII, spam, prompt injection. No data leaves your server. MIT license. Drop-in.

Community moderation UX

Borrow pattern SAFEskies

Role-based moderation roles (admin / moderator), transparent moderation logs, community-controlled feed curation. AT Protocol dependency — adapt the logic, not the code.

Community-scale decisions

Integrate via API Loomio

Consensus-oriented decision-making. Discussion → proposal → vote. Cooperative-owned, AGPL. REST API integration from dyad's SvelteKit frontend.

Institutional governance

Integrate via API Decidim

Full civic engagement suite: assemblies, proposals, budgets, elections, accountability. GraphQL API. AGPL. Run as a standalone service.

Pre-decision sense-making

Fork upstream Polis

Opinion clustering, large-group deliberation. Use as the layer before formal proposals. Fork upstream Polis only — not Blacksky's AT Protocol fork.

Civic participation

Skip Consul Democracy

Overlaps entirely with Decidim. Less modular, less actively maintained. Decidim covers every use case Consul does, with better documentation.

Content moderation · Automated layer

LocalMod Fork it

MIT license Python 116 stars REST API
Content moderation

This is the right answer for dyad's automated moderation layer. LocalMod runs entirely on your own infrastructure — no content ever leaves your server. That is not a configuration option; it's the architecture.

What it classifies, out of the box: toxicity (0.75 balanced accuracy, above Amazon Comprehend and Google Perspective), NSFW text and images, PII detection (1.0 accuracy in benchmarks), spam (0.99 F1), and prompt injection (0.89 F1).

Call it from a SvelteKit API route on content submission. Flag → route to human moderators if needed. The pipeline is: user posts → LocalMod API → severity score → auto-suppress or moderator queue.

Governance

Not applicable. LocalMod is a classification tool, not a decision-making platform. It tells you what content might be harmful; it does not decide what to do with it. That decision layer belongs to community moderators using a role-based interface.

Pair it with the SAFEskies UX pattern (see below) for the human moderation side of the stack.

Integration path

Self-host as a Python API server. dyad's SvelteKit API routes call it on content submission. ~3.5GB of local ML models. One pip install. MIT license — no open-source obligations.

Content moderation · Community roles

SAFEskies Borrow the pattern

MIT license Next.js + React Community feeds
Content moderation

SAFEskies implements a clean role-based moderation system: admin (full control), moderator (review and act on flagged content), member (flag, report). Transparent moderation logs. Community-controlled curation.

The logic is exactly what dyad needs for the human-in-the-loop layer that sits on top of LocalMod's automated flagging.

The problem: SAFEskies is built entirely on AT Protocol for identity. Since dyad isn't using AT Protocol, the code can't be directly forked without stripping its entire auth layer. Study the design and decision model; reimplement the role architecture in dyad's SvelteKit + Supabase stack.

Governance

SAFEskies is not a governance platform. It handles moderation roles, not community decisions. The two are complementary — moderation determines what content survives; governance determines what the community does.

Integration path

Don't fork. Study the admin UX and role model. Build the equivalent natively in Svelte with Supabase RLS backing the permission tiers (admin / moderator / member).

Governance · Community-scale decisions

Loomio Integrate via API

AGPL-3.0 Ruby on Rails 2,547 stars REST API
Content moderation

Not a moderation tool. Loomio is decision infrastructure — it governs what communities decide to do, not what content is allowed. The only adjacent use is community-level policy decisions: "should we adopt a code of conduct?", "who should be our next moderator?" That's governance, not moderation.

Governance

This is Loomio's entire purpose, and it's very good at it. The core model is discussion → proposal → vote, deliberately consensus-oriented rather than majority-rule. A proposal surfaces objections; the group reaches a workable decision rather than steamrolling a minority.

Models: group, discussion, poll (consent, ranked choice, time poll, score), membership, subgroups. Designed to feel like a conversation that reaches a decision — not a formal ballot.

Important: Loomio is itself a worker cooperative. Its governance philosophy matches dyad's steward-ownership model precisely. This is not just a technical fit; it's a values fit.

Integration path

Run Loomio as a separate service. dyad's SvelteKit frontend calls it via REST API. Communities using dyad for in-person coordination use Loomio for the decisions that coordination produces. AGPL obligation: publish any modifications to Loomio itself if you run it as a web service.

Governance · Institutional scale

Decidim Integrate via API

AGPL-3.0 Ruby on Rails 1,764 stars GraphQL API
Content moderation

Decidim has basic moderation built in — report content, hide content, moderator dashboard. It's adequate for formal civic contexts but not a replacement for LocalMod's automated classification. Think of it as moderation-of-record for proposals and debates, not real-time content filtering.

Governance

The most comprehensive platform in this assessment. 28 modules covering everything a formally governed community needs:

Assemblies — permanent participatory bodies with members and working groups. Proposals — submit, amend, endorse, vote, track implementation. Meetings — agenda, minutes, linked proposals. Budgets — participatory budgeting with real allocation. Elections — formal voting with auditability. Blogs — community editorial publishing. Collaborative texts — collective document editing. Accountability — track whether commitments were kept.

Used by Barcelona, Helsinki, the French government, and dozens of European cities. Decade of production use.

Integration path

Run as a standalone service. dyad's SvelteKit frontend talks to decidim-api (GraphQL). Communities never see "Decidim" in the UI — it surfaces through dyad's design. AGPL applies: publish modifications to Decidim itself.

Governance · Pre-decision sense-making

Polis Fork upstream

AGPL-3.0 JS + Clojure Opinion clustering
Content moderation

Not applicable. Polis maps opinion space; it does not classify or filter content.

Governance

Polis sits before formal governance, not inside it. Large groups, contested topics, no moderator-curated agenda — people vote on statements, Polis clusters the result to show where rough consensus already exists and where divisions are real.

The sequence: Polis (what does our community actually think?) → Loomio or Decidim (what do we formally decide?). Without Polis, communities jump straight to proposals that lack a shared foundation. With it, proposals start from known ground.

Note: Fork upstream Polis, not Blacksky's fork. Blacksky added AT Protocol identity into their fork (the Delphi pipeline). Take the upstream codebase only — and optionally pull Blacksky's Python clustering service as a standalone component, which can run without AT Protocol.

Integration path

Run as a separate service. Embed Polis conversations into dyad community pages via API. The math/clustering service (UMAP + HDBSCAN + sentence embeddings) is the most valuable extractable component.

Civic participation · Skipped

Consul Democracy Skip

AGPL-3.0 Ruby on Rails 1,533 stars
Why not

Consul overlaps almost entirely with Decidim — petitions, proposals, debates, budgets, polls. Originally built for Madrid city government.

Decidim is more modular, more actively maintained, has better documentation, a GraphQL API, and is deployed across more European contexts. There is no meaningful advantage to Consul for dyad's use case. Running both would be redundant.

If you need one Rails-based civic governance platform: choose Decidim.

What it does have

Consul's codebase has strong petition and signature-threshold logic that Decidim's decidim-initiatives module replicates. If you want to study how signature-threshold civic initiatives work in a simpler codebase, Consul is readable. But don't run it as a service.

Recommendation

The full modular stack.

Rather than forking one monolithic platform, assemble purpose-built tools. Each layer does one thing well. dyad's SvelteKit frontend is the coherent surface that ties them together.

A decade of civic technology development, assembled. dyad's job is to make it coherent, local, in-person, and owned by the people using it.
