# Simplicity reviewer must never cut explicit user requirements

## Problem

The code-simplicity-reviewer agent was used to review the Session 3 plan. It recommended deferring 7 items that were **explicit user requirements**:

1. **copy.ts** — co-founder independence requirement ("changing a word should mean editing one file")
2. **Landing page discover embed** with waitlist/login modals
3. **Progressive slot disclosure** — teaser before response, full slots after
4. **Mobile Playwright tests** — required for alpha test coverage
5. **Browser `confirm()` replacement** — user explicitly said "NEVER USE CONFIRM()"
6. **Stack of cards profile** — specific design pattern from design reference branch
7. **Waitlist modal** on landing page

The simplicity reviewer's rationale was "defer to v0.2" or "alpha doesn't need this." But these were not optional enhancements — they were stated requirements from the product owner.

This caused a significant disruption: the user discovered the cuts, had to reinstate them manually, and lost trust in the review process.

## Root Cause

The simplicity reviewer optimizes for **minimum code** but has no awareness of **requirement provenance**. It treats all plan items equally — a nice-to-have and a hard requirement look the same in a markdown checkbox. It applied YAGNI heuristics to items that had already passed the "do we need this?" gate.

## Solution

### For Claude (when running review agents):

**The simplicity reviewer's scope is HOW, never WHAT.** It can simplify implementation approaches (e.g., "use sequential calls instead of an RPC") but must never recommend removing or deferring a feature/requirement.

When synthesizing review findings:
- **Discard any simplicity finding that says "defer" or "remove" a plan item** unless you can verify it was NOT an explicit user requirement
- If uncertain, keep the item and flag it for user triage rather than silently cutting it
- The simplicity reviewer's role is: "given that we're building X, here's a simpler way to build X" — NOT "we shouldn't build X"

### For plan documents:

Mark explicit requirements distinctly from nice-to-haves. Use language like:
- `[REQUIRED]` or `[USER REQUIREMENT]` for items the user explicitly stated
- `[NICE-TO-HAVE]` for items generated by analysis/review agents

This gives the simplicity reviewer a signal about what it can and cannot cut.

## Prevention

- **After running the simplicity reviewer on a plan**: diff the plan before and after, and verify no requirement items were removed
- **Include this instruction when invoking the simplicity reviewer**: "Simplify HOW items are implemented, never cut WHAT is being implemented. All plan items are requirements unless marked otherwise."
- **Save user requirements in memory**: when a user states a requirement emphatically, save it to memory so it persists across conversations and review cycles
